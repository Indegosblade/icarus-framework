# Parser Ecosystem

The parser ecosystem manages, validates, and tests parsers at scale.

## Components

### 1. YAML Manifests

Every parser ships with a co-located YAML manifest validated by JSON Schema at load time. The manifest declares:

- **Identity** — parser_id, version, spec_version
- **Quality** — quality_tier (production/candidate/prototype/private), reliability (Admiralty A-F), default_confidence
- **Detection** — specificity_level (1-100), identification markers
- **Capabilities** — consumes (input types), produces (entity types)
- **Testing** — fixtures directory, golden output path

Schema: `icarus/parsers/schema/parser_manifest.schema.json`

### 2. Parser Registry

The registry holds all parser instances and their manifests. Registration is automatic: every concrete `BaseParser` subclass found anywhere under `icarus/parsers/` — including local-only parsers in the gitignored `icarus/parsers/private/` package — is discovered and registered at import time, and an installed distribution can advertise additional parsers through the `icarus.parsers` entry-point group. There is no hand-maintained list of parsers to keep in sync; discovery and manifest-load failures are logged rather than silently swallowed. See [about/PARSERS.md](../about/PARSERS.md) for the registration walkthrough.

When `icarus build` runs without `--parser`, the registry runs a **detection contest**:

1. Every registered parser's `identify()` runs against the source
2. Parsers that return True are candidates
3. Candidates sorted by specificity (lower wins), then confidence (higher wins)
4. Winner becomes the active parser

This means CloudTrail (specificity 5) always beats Windows (20) which always beats generic (100). If you write a parser with specificity 10, it automatically slots between CloudTrail and Windows.

### 3. Two-Tier Catalog

- `parsers.json` — production parsers (ships with the framework)
- `parsers-devel.json` — candidate parsers (under evaluation)

Generated by `icarus/parsers/catalog/update_catalog.py` which walks all YAML manifests.

### 4. Test Harness

4 mandatory quality gates:

| Gate | What It Checks |
|------|---------------|
| **Golden output** | Entity counts match saved baseline (tests/golden/*.json) |
| **Idempotency** | Second run over same data adds zero new entities |
| **Schema conformance** | Parser only writes to tables declared in its manifest |
| **Zero-PII** | HYGEIA verify_clean passes on the output database |

```bash
icarus parser test windows
```

All 8 production parsers pass all 4 gates; the 1 candidate parser is under evaluation.

## Quality Tiers

| Tier | In Catalog | Requirements |
|------|:----------:|-------------|
| `production` | parsers.json | All 4 harness tests pass, manifest validates, real-world validation |
| `candidate` | parsers-devel.json | Under evaluation, may have known issues |
| `prototype` | No | Proof of concept |
| `private` | No | Internal use only |
